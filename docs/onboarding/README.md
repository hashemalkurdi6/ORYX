# Onboarding

Onboarding documents for new contributors — Armen, future hires, and future-you after a long break.

Each doc is its own file, named after the audience (e.g. `armen.md`). When a new person joins, write them their own file rather than editing an existing one — the contents need to assume their starting point.

## What an onboarding doc should cover

- Project overview in one paragraph (what is ORYX, who is it for).
- Tech stack and where things live.
- How to run things locally.
- Tooling we use (Claude Code agents, design system).
- Where to find the spec, the audit, decisions, design tokens.
- Conventions: commits, decisions, prompts.
- Current launch timeline.
- Who to ask about what.

When the doc references things outside `docs/`, link to them by relative path so the reader can click straight through in Obsidian.
