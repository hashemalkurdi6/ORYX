# Alembic migrations

This project uses Alembic for schema migrations going forward. Legacy schema
was created by `_USER_COLUMN_MIGRATIONS` in `app/main.py` (still runs on
startup for the existing column list — do not remove).

## Existing databases (pre-Alembic)

Any environment that already has tables created via `Base.metadata.create_all`
and `_USER_COLUMN_MIGRATIONS` must be stamped once so Alembic does not try to
re-create those tables:

```bash
alembic stamp head
```

After that, pull new branches and run:

```bash
alembic upgrade head
```

## Fresh databases

```bash
alembic upgrade head
```

The baseline revision is a no-op (schema is assumed to already exist OR to be
created by the legacy startup SQL). New features create real migrations on top
of the baseline.

## Authoring migrations

```bash
alembic revision -m "describe_change"                 # empty scaffold
alembic revision --autogenerate -m "describe_change"  # diff models vs DB
alembic downgrade -1                                  # roll back one step
```

Review every autogenerated migration — especially around JSON/JSONB type
changes and index names — before committing.
